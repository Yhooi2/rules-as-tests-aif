# install-self-verification — single-PR implementation kickoff

> **Type:** single buildable implementation kickoff (NOT a meta-launch / umbrella index — no Stage gates, no Sub-wave table, no Launch table). Dispatch this file directly to aif; it produces code + one PR to `staging`.
> **Authoritative for:** the install-time self-verification capability (fences fire / shields block) — scope, deliverables, acceptance. NOT authoritative for project goal — see [README.md#why-this-exists](../../../README.md#why-this-exists).
> **Base:** `staging`. **Stack of record:** `react-next` (richest custom-rule set: R2/R7/R8 + R12/R14/R20). **$0-in-CI, deterministic, no network needed in the container.**

## 0. One-line goal

Make `./setup -y --full` **prove** — not just assert presence — that the fences it built **FIRE on bad input** and the shields it raised are **wired & active**, expose this as a repeatable consumer gate `npm run check:fences-fire` (+ `check:shields-up`) wired into `npm run validate`, have the installer **self-run** the proof at the end of a `--full` install and print a firing summary, AND **mutation-verify the GENERATED tests** — the first run after install, over EVERY generated rule, prove each generated paired-negative test actually KILLS mutants of its generated rule (not test-theatre), via a deterministic install-time gate (plus an on-demand deep pass and a self-contained CI proof on the fixture rules).

## 1. Why (the verified gap — evidence)

The one-button pipeline is already built and shipping on `staging` (`fef73c20b`): 4-stack detect, live-research adapter (#805), fences (`40-configs.sh`: eslint custom-rule plugin → `eslint-rules-local/`, depcruise, stryker, CI), shields (`50-hooks.sh`: husky pre-commit/pre-push + `core.hooksPath=.husky`), and a consumer `validate` of 9 gates (`setup.d/70-deps.sh:68`).

**The single unbuilt thing:** the installer checks **presence/binding**, never **behaviour** — as a *shipped, multi-rule, consumer-runnable* capability.
- No installer layer and no consumer gate feeds a deliberately-bad sample to a rule and asserts non-zero exit. The RuleTester firing tests (`*.test.ts`) are deliberately **stripped** from the consumer payload (`setup.d/40-configs.sh:115`, repeated `:125`/`:136`) and live next to the rules in `eslint-rules-local/`, not under the consumer's `src/**` — so no firing test reaches or runs in the consumer.
- **Relationship to `tests/install-sh/f17-lint-rules-planted-violation.test.sh`:** f17 already proves R2 (`no-unsafe-zod-parse`) fires at install for ONE rule, via the ESLint `Linter` API — but it is a **repo-side CI test**, not a shipped consumer `npm run` gate, and it covers one rule. This umbrella **generalizes** f17 into a shipped, multi-fence, consumer-runnable gate. Reuse f17's proven mechanism (see §D1).
- `check:globs` proves a rule's globs match ≥1 file (not inert); `check:enforced` proves R2 is bound on boundary files. **Neither proves the rule REJECTS bad code.** `lint` runs ESLint on the consumer's *clean* code → passes whether or not the rule actually fires.
- `50-hooks.sh:69` sets `core.hooksPath` but nothing proves a bad commit/push is blocked; `99-finalize.sh:196` delegates that to a manual human Next-step.

This is exactly the operator's stated requirement: «в конце — что все работает и все падает когда надо и все ловит» / «установщик обязательно сам проверяет и тестирует что он все правильно настроил и что все щиты подняты и все заборы построены».

## 3. Deliverables

### D1 — `check:fences-fire` (the heart: rules fail-when-they-should)

A new dependency-free bash gate `packages/core/audit-self/check-fences-fire.sh`, shipped to the consumer exactly like its siblings:
- Add a `copy_safe` + `chmod_safe +x` for it in `setup.d/40-configs.sh` next to the `check-rule-globs.sh` / `check-rule-enforced.sh` copies (the precedent block at `40-configs.sh:~20-40`). Consumer path: `scripts/check-fences-fire.sh`.
- **Firing mechanism — REUSE the proven f17 technique, NOT the eslint CLI.** `tests/install-sh/f17-lint-rules-planted-violation.test.sh:32-36,80-149` documents that `eslint <file>` with the real flat config crashes (`ERR_REQUIRE_CYCLE_MODULE` / `ERR_UNKNOWN_FILE_EXTENSION ".ts"`). It instead drives the ESLint **`Linter` API via tsx**, locates the framework's own binary (`$REPO_ROOT/node_modules/.bin/eslint`, `/app/node_modules/.bin/eslint`) and symlinks that `node_modules` into the scratch consumer, and **SKIPs cleanly** when no eslint is resolvable. The probe MUST follow this: resolve eslint via local-bin → `npx --no-install` (never a fresh `npm install` — the container has no npm-registry network), run via the `Linter` API (or `npx --no-install eslint` only if a local bin is present), and SKIP-with-guidance (rc=0) when eslint is absent — mirroring `check-rule-enforced.sh:86-97` for the resolution half.
- **Two fence classes (react-next has both):** each fixture declares the `expect-rule-id` it must trigger.
  - **Class 1 — standalone module rules** (id = the rule itself): `no-unsafe-zod-parse` (R2, core), `no-direct-time-randomness` (R7, core, opt-in), `require-otel-span` (R8, core, opt-in), `no-server-imports-in-client` (R12, next preset). On the bad fixture eslint reports *that* id.
  - **Class 2 — declarative recipes** enforced via the wrapper rule `rules-as-tests/restricted-syntax-audit-exempt` (this is how R14/R20 + the 10 `appliesTo:["next"]` recipes are enforced after they were migrated out of standalone rules — see `packages/preset-next-15-canonical/eslint-rules/index.ts:3-14`). On the bad fixture eslint reports `restricted-syntax-audit-exempt` (assert that id + that the message names the recipe). Ship at least one Class-2 fixture so the declarative tier is proven to fire too.
  - Do **NOT** assert R14/R20 report `require-form-safe-parse`/`require-use-server-directive` — those standalone rules no longer exist; the wrapper id fires instead.
- **Fixtures:** ship a curated bad/good pair per covered fence under `packages/core/audit-self/fixtures/fences-fire/<short-id>.bad.<ext>` + `.good.<ext>` (the `audit-self/fixtures/` dir exists). Copy the whole `fences-fire/` subtree to the consumer (`scripts/fences-fire-fixtures/`) in `40-configs.sh`. Each `.bad` MUST be the minimal snippet the fence is designed to reject (cross-checked per T-ISV-A); each `.good` MUST be the closest-passing counterpart (true paired-negative, not an unrelated clean file).
- **Algorithm:** iterate the shipped fixtures; for each, resolve whether its `expect-rule-id` is **enabled** in the consumer's applied config (`eslint --print-config` per `check-rule-enforced.sh`). If enabled: bad fixture MUST exit non-zero AND surface `expect-rule-id` (fires; exit 0 ⇒ FAIL «fence did NOT fire — silently inert»); good fixture MUST exit 0 (non-zero ⇒ FAIL «false-positive»). Enabled-but-no-fixture ⇒ WARN (coverage gap, list it); fixture-but-rule-disabled ⇒ skip silently.
- **R7/R8 nuance:** opt-in behind `AIF_STRICT_RUNTIME=1`. Treat «not enabled in applied config» as *skip*, never *fail* — only enabled fences are required to fire.

### D5 — mutation-verify ALL the GENERATED tests, first run after install (the deepest catch — operator-mandated)

`check:fences-fire` proves a rule fires on ONE bad fixture; it does NOT prove the **generated** paired-negative test (produced by the live-research/synthesizer, #805) is *rigorous*. A generated test that passes but would not catch a *broken* generated rule is test-theatre. **Operator mandate (binding):** after the rules are written and their tests generated, the installer must check them for theatre — **the first time, right after installation — over EVERY generated rule, not a demo.** Mutation testing is the canonical tool for exactly this (it measures whether a test catches a deliberate fault), and it is already this project's code theatre-detector — **ADAPT the `run-bash-mutation.sh` discipline (SSOT #91), do NOT build a new mutation engine.**

- **Enumerate ALL generated rules from the emitted manifest; each carries its paired-negative inline.** `emit()` (`emit.ts:17-44`) writes the manifest to the **exact path `${PROJECT_ROOT}/.ai-factory/synthesizer-output/rules-manifest-additions.json`** (`install()` → `emit()`, path = `installer/install.ts:18` `OUTPUT_SUBPATH=['.ai-factory','synthesizer-output']`, `:26-27` `outputDirOf`, `:136` `emit(plan,outputDir)`) = `{<ruleId>: {...rule, research}}`, carrying each generated rule's `selector`, `examples:{bad,good}`, and **`negative-test`** (`{input, "expect-violation"}` — note the *hyphenated* key on `SynthesizedRule`, `types.ts:54-61`; the upstream `GenerateClient` port uses `negativeTest`, `generate-port.ts:31` — handle both spellings). Its sibling **`rules-lock.json` is in the SAME dir** (`install.ts:108`) and carries `ruleIds` (`installer/types.ts:31`) for the cross-check. The gate reads selector + negative-test for each rule directly from that manifest. NOTE: `eslint-rules-snippet.json` carries only selector+message (NO negativeTest), and `selection.json` is an intermediate — so the **manifest at the path above is the gate's source**, not the eslint snippet, not the consumer root, not `.ai-factory/rules-research/` (that is the *input* dir). No separately-shipped `*.test.ts` is needed (the §1 stripped-`*.test.ts` concern does NOT apply — the negative-test travels *inside* the manifest). **Manifest absent at that path ⇒ treat as zero generated rules → degrade with guidance (rc=0), NEVER fabricate a pass** — the manifest exists only when `${PROJECT_ROOT}/.ai-factory/rules-research/<stack>.{research,selection}.json` were pre-authored by the human-run rule-research/MCP protocol (no `setup.d/*.sh` writes them — `80-rule-bootstrap.sh:41-52` degrades to zero rules without them; so on a vanilla `--full` with no research artefacts the gate degrades, it does NOT pass-vacuously).
- **The generated rules are DECLARATIVE — frame mutation as selector perturbation.** The synthesizer emits each as a **declarative `no-restricted-syntax` selector** (`fixtures/no-head-element.selection.json` → `"selector":"JSXOpeningElement[name.name='head']"`, `"presence":"forbid"`; `withManualDrop` drops any non-declarative `manual` candidate before ship). So there is **no imperative rule module to Stryker-mutate** — Stryker (TS-AST) is the WRONG engine (SSOT #91 notes «Stryker covers TS only»). The mutation target is the **selector**: perturb it (`name.name='head'`→`'xhead'`, drop the attribute constraint, swap the node type).
- **Per generated rule:** read its `selector` + `negativeTest` (bad must violate `expect-violation`, good must pass); apply the deterministic selector perturbations; assert each perturbation makes the «bad violates» assertion BREAK (mutant KILLED) ≥ the **≥60% floor** (reuse `run-bash-mutation.sh:41` `MIN_KILL=${3:-60}` + audit §A.4). A perturbation the negativeTest still passes ⇒ that generated test is theatre ⇒ gate FAILS, naming the rule.
- **Run it FIRST TIME AFTER INSTALL (the operator-mandated placement).** Wire the gate into the installer self-run capstone (D3 — `setup.d/99-finalize.sh` / `90-verify.sh`), **after** `80-rule-bootstrap` generated the rules, so a `--full` install ends with `✓ self-verify: … M/M generated tests killed their mutants (≥60%)`. This deterministic, dependency-light perturbation engine (NOT pip-universalmutator — may be absent on consumer/container, so it is fast enough for a one-shot post-install run) is the must-run. Degrade gracefully (skip + guidance) when there are **zero** generated rules (own-repo / no live-research artefacts → `ruleIds=[]`) or eslint/deps absent; MUST NOT run on the CI self-install path (FULL unset).
- **`packages/core/synthesizer/run-generated-rule-mutation.sh` (on-demand local DEPTH):** mirrors `run-bash-mutation.sh`: ADAPT the **universalmutator custom-operator pattern** (SSOT #91 — same engine the bash gate uses) with selector operators, kill-rate vs the same ≥60% floor. **Delivery = SESSION-BOUND / LOCAL, NOT a CI gate** (mutation is devDep/on-demand per the `run-bash-mutation.sh` header; README «CI = last resort»). Consumer surface `test:mutation:generated`. NOT added to `validate`. This is the deeper breadth pass; the install-time deterministic gate above is the must-run.
- **CI proof-of-function (obligatory) — SELF-CONTAINED, over MULTIPLE fixture rules.** `packages/core/synthesizer/run-generated-rule-mutation.test.ts` — vitest, does NOT shell to universalmutator/pip (local-only / skips in CI — `run-bash-mutation.test.ts:9-13`). Build the demo rules deterministically from fixtures and prove the gate iterates **>1** rule: `no-head-element` via `synthesizeGenerate(noHeadResearchPlan, stubGenerateHead)` where `noHeadResearchPlan = JSON.parse(fixtures/no-head-element.research.json)` — a real `ResearchPlan` (arg-1's actual type, `generate.ts:20-23`); do **NOT** pass `no-head-element.selection.json` (that is the `GenerateClient` *output* shape). `stubGenerateHead` = NEW deterministic `GenerateClient` stub (mirror `stubGenerateForbid` `generate-stubs.ts:115`, used by `generate.test.ts:(d)`) returning ONE candidate with `entryId:'next-no-head-element'` (MUST equal the real id in the plan — else `plan.rules` empty = vacuous green = `T-ISV-B`), `presence:'forbid'`, `selector:"JSXOpeningElement[name.name='head']"`, the `no-restricted-syntax` negativeTest — plus **≥1 more** generated rule to prove multi-rule iteration. **PREFERRED construction:** a SECOND forbid candidate in the SAME `stubGenerateHead` against the SAME plan (`synthesizeGenerate` loops `selection.rules`, `generate.ts:46`, so one stub returning two forbid candidates → a 2-rule `SynthesisPlan` in one call; both `entryId`s MUST be real ids in the plan, else silently skipped — `generate-stubs.ts:113-114`). The 2nd pattern already in `no-head-element.research.json` (`next-server-only-boundary`) is **research-only / not-shipped** → NOT usable as the 2nd forbid; add a 2nd forbid pattern to the fixture (or a tiny purpose-built 2-pattern plan). Avoid pulling in `react-spa`/`react-native` plans + extra stubs (reachable, but needless cross-stack surface for a «iterates >1 rule» proof). Assert: (a) for each rule, every perturbation kills the negativeTest ≥ floor; (b) **recursive self-application** — neuter one generated test ⇒ gate goes RED (`T-ISV-B`). Deterministic, $0, no network → the obligatory CI proof.
- **Scope:** the install-time gate covers EVERY generated rule present in the consumer (`ruleIds`). The CI proof exercises it on the synthesizer fixture rules (`no-head-element` + ≥1 of `error-boundary`/`flatlist`). Real per-stack live-research generated rules can only be exercised on the consumer (no MCP in the container) — the fixture corpus is the container-side proof; state this in the PR.
